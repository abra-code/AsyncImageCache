# AsyncImageCache demo

A small multiplatform (iOS + macOS) SwiftUI app that exercises `CachedImage` in a scrolling `LazyVStack`:
local and remote images side by side, so you can watch the library's behavior in a real UI.

## What it shows
- Off-main decode + downscale (toggle "Cap decoded width to 400 px" and watch large sources shrink without a
  main-thread hitch).
- The placeholder grid preview (a soft gradient) filling the reserved box while pixels load, instead of gray.
- Zero-reflow layout: every row passes `intrinsicSize`, so the box is reserved at the exact aspect before the
  image arrives - nothing jumps on hydration.
- Memory eviction: "Tiny memory budget (2 MB)" swaps in a store with a tiny cost limit, so scrolling evicts
  decoded variants and reloads them (and the placeholder reappears on the reload).
- The two-tier cache: "Clear memory" then scroll to see the disk tier serve instantly; "Clear memory + disk"
  to force a cold reload (locals re-read from the bundle, remotes re-fetch).
- Animated sources: one generated GIF carries an "animated" badge (from `store.isAnimated(for:)`) and renders
  as its flattened first frame - playback is a consumer concern, see ../Private/Animated_GIF_Playback_Analysis.md.

Local images are LARGE gradients (plus one animated GIF) generated at build time; remote images come from
picsum.photos (needs a network connection).

## Seeing the placeholder preview
The blurry placeholder is DERIVED from the decoded pixels, so a never-before-seen remote image has no preview
yet and shows flat gray - it appears only once the image has loaded once (then it survives relaunch via the
on-disk xattr). To see it clearly: let the images load, then turn on "Show placeholders only" - each row that
has loaded once renders its cached blurry preview statically (via `store.placeholderImage(for:)`). The tiny
memory budget also surfaces it, since evicted variants briefly show the placeholder again on reload.

## Build and run
This project is generated by [XcodeGen](https://github.com/yonyz/XcodeGen) from `project.yml`.
To regenerate, cd to this `Demo/` directory and:

```sh
xcodegen generate
open AsyncImageCacheDemo.xcodeproj
```

Then pick the `AsyncImageCacheDemo-iOS` or `AsyncImageCacheDemo-macOS` scheme and run. (Set your signing team
in the target's Signing & Capabilities on first run.)

## How the "no big blobs in git" part works
`Scripts/generate-images.swift` renders the local images into `GeneratedResources/` during a pre-build phase.
That folder's contents are gitignored (only a `.gitkeep` is committed), so nothing large lands in the repo;
the images are recreated on the next build. The step is idempotent - it skips files that already exist.

The generator is invoked as `xcrun --sdk macosx swift ...` so it runs against the macOS host SDK even when the
iOS target is building, and `ENABLE_USER_SCRIPT_SANDBOXING` is off so it may write into the source tree.
